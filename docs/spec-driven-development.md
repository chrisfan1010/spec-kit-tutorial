# 规范驱动开发 (SDD)

> 规范驱动开发 (Spec-Driven Development, SDD) 颠覆传统软件开发，让规格成为代码的源头。

## 权力反转

数十年来，代码是王者。规格为代码服务——它们只是我们搭建后丢弃的脚手架，一旦"真正"的编码工作开始就被丢弃。我们编写 PRD 来指导开发，创建设计文档来告知实现，绘制图表来可视化架构。但这些总是从属于代码本身。代码就是真理，其他一切都不过是美好的意图。代码是真理的来源，随着代码推进，规格很少能跟上。当资产（代码）和实现是一体的很难在不尝试从代码构建的情况下进行并行实现。

规范驱动开发 (SDD) 改变了这种权力结构。规格不为代码服务——代码为规格服务。产品需求文档 (PRD) 不是实现的指南；它是生成实现的源头。技术计划不是告知编码的文档；它们是产生代码的精确定义。这不是对软件开发方式的渐进改进。这是对什么驱动开发的根本反思。

规格与实现之间的差距自软件开发诞生以来就困扰着我们。我们尝试用更好的文档、更详细的需求、更严格的流程来弥合它。这些方法失败是因为它们承认差距是不可避免的。它们试图缩小它，但从未消除它。SDD 通过使规格及其具体实现计划可执行来消除差距。当规格和实现计划生成代码时，没有差距——只有转换。

这种转换现在成为可能，因为 AI 可以理解并实现复杂的规格，并创建详细的实现计划。但没有结构的原始 AI 生成会产生混乱。SDD 通过精确、完整且明确的规格和后续实现计划提供这种结构，使它们能够生成可工作的系统。规格成为主要产物。代码成为其在特定语言和框架中的表达（作为实现计划的实现）。

在这个新世界中，维护软件意味着演进规格。开发团队的意图以自然语言表达（**意图驱动开发**）、设计资产、核心原则和其他指导方针。开发的**通用语言**提升到更高层次，代码是最后一英里的方法。

调试意味着修复生成错误代码的规格及其实现计划。重构意味着为清晰度而重组。整个开发工作流围绕规格作为真理的中心来源进行重组，实现计划和代码作为不断重新生成的输出。使用新功能更新应用程序或创建新的并行实现（因为我们是创造性生物）意味着重新访问规格并创建新的实现计划。因此，这个过程是 0 -> 1, (1', ..), 2, 3, N。

开发团队专注于他们的创造力、实验和批判性思维。

## SDD 工作流实践

工作流从一个想法开始——通常模糊且不完整。通过与 AI 的迭代对话，这个想法变成了全面的 PRD。AI 询问澄清问题，识别边缘情况，并帮助定义精确的验收标准。在传统开发中可能需要数天会议和文档的工作，在数小时的专注规格工作中就能完成。这改变了传统 SDLC——需求和设计成为持续的活动而不是离散的阶段。这是一个**团队流程**，团队审查的规格被表达和版本化，在分支中创建，并合并。

当产品经理更新验收标准时，实现计划会自动标记受影响的技术决策。当架构师发现更好的模式时，PRD 会更新以反映新的可能性。

在整个规格过程中，研究代理收集关键上下文。他们调查库兼容性、性能基准和安全影响。自动发现并应用组织约束——您公司的数据库标准、身份验证要求和部署策略无缝集成到每个规格中。

从 PRD 中，AI 生成将需求映射到技术决策的实现计划。每个技术选择都有文档化的理由。每个架构决策都可以追溯到特定的需求。在整个过程中，一致性验证不断提高质量。AI 分析规格的模糊性、矛盾和差距——不是作为一次性门，而是作为持续改进。

代码生成在规格及其实现计划足够稳定时就开始，但它们不必是"完整的"早期的生成可能是探索性的——测试规格在实践中是否有意义。领域概念成为数据模型。用户故事成为 API 端点。验收场景成为测试。这通过规格合并开发和测试——测试场景不是在代码之后编写的，它们是生成实现和测试的规格的一部分。

反馈循环延伸到初始开发之外。生产指标和事件不仅触发热修复——它们为下一次重新生成更新规格。性能瓶颈成为新的非功能需求。安全漏洞成为影响所有未来生成的约束。规格、实现和运营现实之间的迭代舞蹈是真正理解出现的地方，传统 SDLC 在其中转变为持续演进。

## 为什么 SDD 现在很重要

三个趋势使 SDD 不仅成为可能而且是必要的：

首先，AI 能力已达到阈值，自然语言规格可以可靠地生成工作代码。这不是要取代开发人员——而是通过自动化从规格到实现的机械翻译来提高他们的效率。它可以放大探索和创造力，轻松支持"重新开始"，并支持加法、减法和批判性思考。

其次，软件复杂性持续呈指数增长。现代系统集成数十种服务、框架和依赖项。通过手动流程使所有这些与原始意图保持一致变得越来越困难。SDD 通过规格驱动的生成提供系统性对齐。框架可能会发展以提供 AI 优先支持，而不是人类优先支持，或者围绕可重用组件进行架构。

第三，变化速度加快。今天的需求变化比以往任何时候都快。转变不再特殊——这是预期的。现代产品开发需要基于用户反馈、市场条件和竞争压力的快速迭代。传统开发将这些变化视为障碍。每次支点都需要手动将更改传播到文档、设计和代码。结果是缓慢、仔细的更新限制速度，或者快速、不计后果的变化累积技术债务。

SDD 可以支持假设/模拟实验："如果我们需要重新实现或改变应用程序以促进销售更多 T 恤的业务需求，我们如何为此实现和实验？"

当规格驱动实现时，转变成为系统化的重新生成而不是手动重写。更改 PRD 中的核心需求，受影响的实现计划会自动更新。更改用户故事，相应的 API 端点会重新生成。这不仅仅是关于初始开发——而是通过不可避免的变化保持工程速度。

## 核心原则

**规格作为通用语言**：规格成为主要产物。代码成为其在特定语言和框架中的表达。维护软件意味着演进规格。

**可执行规格**：规格必须足够精确、完整和明确，以生成可工作的系统。这消除了意图和实现之间的差距。

**持续改进**：一致性验证持续发生，而不是作为一次性门。AI 持续分析规格的模糊性、矛盾和差距。

**研究驱动的上下文**：研究代理在整个规格过程中收集关键上下文，调查技术选项、性能影响和组织约束。

**双向反馈**：生产现实通知规格演进。指标、事件和运营学习成为规格改进的输入。

**分支探索**：从相同规格生成多个实现方法，以探索不同的优化目标——性能、可维护性、用户体验、成本。

## 使用命令简化 SDD

SDD 方法通过三个强大的命令显著增强，这些命令自动执行规格 → 计划 → 任务工作流：

### `/speckit.specify` 命令

此命令将简单的功能描述（用户提示）转换为完整的结构化规格，并自动进行仓库管理：

1. **自动功能编号**：扫描现有规格以确定下一个功能编号（例如 001、002、003）
2. **分支创建**：从您的描述生成语义分支名称并自动创建它
3. **基于模板的生成**：使用您的需求复制和自定义功能规格模板
4. **目录结构**：为所有相关文档创建正确的 `specs/[branch-name]/` 结构

### `/speckit.plan` 命令

一旦存在功能规格，此命令创建全面的实现计划：

1. **规格分析**：读取并理解功能需求、用户故事和验收标准
2. **宪法合规性**：确保与项目宪法和架构原则一致
3. **技术转换**：将业务需求转换为技术架构和实现细节
4. **详细文档**：生成数据模型、API 合约和测试场景的支持文档
5. **快速启动验证**：生成包含关键验证场景的快速启动指南

### `/speckit.tasks` 命令

创建计划后，此命令分析计划和相关设计文档以生成可执行任务列表：

1. **输入**：读取 `plan.md`（必需）和（如果存在）`data-model.md`、`contracts/` 和 `research.md`
2. **任务派生**：将合约、实体和场景转换为特定任务
3. **并行化**：标记独立任务 `[P]` 并概述安全的并行组
4. **输出**：在功能目录中写入 `tasks.md`，准备好由任务代理执行
